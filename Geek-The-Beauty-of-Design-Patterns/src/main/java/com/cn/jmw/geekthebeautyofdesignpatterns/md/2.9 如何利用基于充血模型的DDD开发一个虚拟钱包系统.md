# 国内充血模型标杆 https://github.com/alibaba/COLA
# 国际DDD领域模型充血模型的圣典 https://github.com/citerus/dddsample-core

充血模型 = 智能机器人： 实体类不仅包含数据，还包含了属于自己的业务规则和行为。
它拥有极强的“自我保护意识”，谁也不能随便通过 set 方法修改它的状态，必须通过它暴露出来的明确业务指令（方法）来操作。

假设我们有一个极其常见的场景：用户修改密码。
业务边界规则是：新密码不能和旧密码相同，且修改后要更新修改时间。

❌ 贫血模型（你我平时写的代码）
在贫血模型中，User 实体毫无灵魂：

Java
// 实体类：只是一堆数据的容器 (提线木偶)
@Data // 自动生成了所有的 get/set
public class User {
private Long id;
private String password;
private LocalDateTime updateTime;
}

// 牵线人：Service 包揽了一切逻辑
public class UserService {
public void changePassword(Long userId, String newPassword) {
User user = userMapper.selectById(userId);

        // 1. 业务校验逻辑在 Service 里
        if (user.getPassword().equals(newPassword)) {
            throw new RuntimeException("新旧密码不能相同");
        }
        
        // 2. 状态修改逻辑在 Service 里
        user.setPassword(newPassword);
        user.setUpdateTime(LocalDateTime.now());
        
        // 3. 落库
        userMapper.update(user);
    }
}
痛点在哪？ 如果明天代码里又多了一个 AdminService 也需要重置用户密码，那个开发人员很可能忘记写“新旧密码不能相同”的校验逻辑，直接调用了 user.setPassword()。边界条件就这么被漏掉了。

✅ 充血模型（真正的面向对象设计）
在充血模型中，我们把属于“用户”的逻辑，还给“用户”自己：

Java
// 实体类：拥有自我管理能力 (智能机器人)
public class User {
private Long id;
private String password;
private LocalDateTime updateTime;

    // 严禁随便暴露 Setter！状态由自己控制

    // 暴露明确的业务行为方法
    public void changePassword(String newPassword) {
        // 1. 实体类自己保护自己的边界规则
        if (this.password.equals(newPassword)) {
            throw new RuntimeException("新旧密码不能相同");
        }
        // 2. 实体类自己维护自己的状态
        this.password = newPassword;
        this.updateTime = LocalDateTime.now();
    }
    
    // 只暴露必要的 Getter 供外部读取
    public String getPassword() { return password; }
}

// Service 退化为“协调者” (Coordinator)
public class UserService {
public void changePassword(Long userId, String newPassword) {
// 1. 获取数据
User user = userMapper.selectById(userId);

        // 2. 发送业务指令 (Service 不再干涉校验细节)
        user.changePassword(newPassword); 
        
        // 3. 落库
        userMapper.update(user);
    }
}
收益在哪？ 逻辑极度内聚！无论谁想改密码，都必须调用 user.changePassword()，那条“新旧密码不能相同”的边界规则永远不可能被绕过或遗漏。你的代码天然就防范了 Bug。


# 后端三层架构 -》 四层架构（接口层、服务层、领域层、基础层）
❌ 传统贫血模型（MVC 三层架构）
在这个结构里，所有的业务逻辑、外部 API 调用、定时任务，最后都会像倒垃圾一样堆进 ReportServiceImpl 里。

```Plaintext
com.company.report
├── controller  (控制层)
│   └── ReportController.java
├── entity      (数据容器，只有 @Data，没有灵魂)
│   └── Report.java
├── mapper      (数据库交互，写 SQL 的地方)
│   └── ReportMapper.java
└── service     (万能的服务层，极其臃肿)
├── impl
│   └── ReportServiceImpl.java  <-- 【灾难现场】包含 if-else 审核逻辑、企业微信 API 调用、EasyExcel 导出代码、甚至 XXL-Job 的任务方法都写在这。
└── ReportService.java
```

✅ 充血模型（DDD 四层标准架构）
现在，我们把上面的系统重构成充血模型。请注意看层次的职责拆分，以及防腐设计（不让具体的框架污染核心业务）。

```Plaintext
com.company.report
│
├── interfaces       (接口层：对外暴露的大门)
│   └── controller
│       └── ReportController.java    <-- 只负责接收 HTTP 请求，参数校验，转交给 Application 层。
│
├── application      (应用层：核心调度员，流程编排)
│   ├── service
│   │   └── ReportAppService.java    <-- 【变薄了】只管调度：开启 @Transactional，捞数据，调实体的方法，调发通知的类。坚决不写业务 if-else。
│   └── job
│       └── ReportAuditJob.java      <-- XXL-Job 定时任务入口，触发 ReportAppService。
│
├── domain           (领域层：系统的灵魂，纯 Java 代码) 🌟🌟🌟 核心变化在这里
│   ├── entity
│   │   └── Report.java              <-- 【充血实体】包含了 approve() 审核动作，状态机流转和业务规则全写在它内部。
│   ├── service
│   │   └── ReportDomainService.java <-- 处理跨多个实体的复杂业务（如有需要）。
│   └── repository
│       └── ReportRepository.java    <-- 【关键思路】只定义接口（如 save()）。领域层不管底层是存 SQL 还是别的地方，它只提需求。
│
└── infrastructure   (基础设施层：打杂的，处理所有技术细节) 🌟 新增的物理隔离区
├── persistence
│   ├── mapper
│   │   └── ReportMapper.java         <-- 真实的 MyBatis/SQL 接口。
│   └── ReportRepositoryImpl.java     <-- 实现 domain 层定义的接口，在这里调用 Mapper 落库。
│
├── external
│   └── WeComNotifier.java            <-- 封装企业微信 API 调用的具体逻辑。
│
└── util
└── ReportExcelExporter.java      <-- 封装使用 EasyExcel 读写文件的具体逻辑。
```
